## Phase 1 — Identity & access: DONE

Two new Django apps in `apps/api`: `accounts` (User, MagicLink) and `projects` (Project, Membership, ShareLink) — schema per Plan.md §8. No passwords anywhere; `django.contrib.auth` stays reserved for Django admin staff logins only. Session-based app identity lives in `request.fl_user`, set by `accounts.middleware.CurrentUserMiddleware` from a `fl_user_id` session key (`core/session.py`).

**Sign-in paths (`accounts/api.py`):**
- **ORCID OAuth2** (`accounts/orcid.py`) — verified against ORCID's live API tutorial before building (2026-07-10). One deviation from Plan.md's literal wording: scope `openid` instead of `/authenticate`, per ORCID's own current guidance that they're equivalent and `/authenticate` is superseded — recorded in `docs/decisions/orcid-scope.md`. Sandbox by default (`ORCID_ENV=sandbox`). State-parameter CSRF protection on the OAuth redirect. Trusts the `orcid`/`name` fields ORCID's token endpoint returns directly (documented behavior, no extra API call).
- **Magic link** (`accounts/magic_link.py`) — signed random token (sha256-hashed at rest, raw token only ever emailed), 15 min TTL, single-use, delivered via Mailpit in dev. Email normalized to lowercase to avoid duplicate accounts.
- **Anonymous** — `POST /auth/anonymous`, optional `display_name`.

**Authorization (`projects/`):** Project/Membership/ShareLink CRUD. `projects.authz.get_authorized_project` returns 404 (not 403) for both "doesn't exist" and "no membership," so unauthorized callers can't distinguish the two. Anonymous users get access **only** by hitting `POST /share-links/{token}/join`, which creates their `User` row and `Membership` together — there's no other path to a Membership for an anonymous kind user. Share-link tokens are sha256-hashed at rest like magic-link tokens (spec names the field `token`; storing it hashed is a deliberate hardening beyond the literal spec, not a behavior change).

**CSRF/sessions:** Django's `CsrfViewMiddleware` doesn't actually protect django-ninja views (ninja marks all its views `csrf_exempt` at the Django level by design, and re-implements the check itself inside its cookie-auth classes) — found this the hard way via a failing test, not by inference. So every state-changing endpoint explicitly carries a cookie-based auth class: `SessionAuth` (requires an existing session — used for the `projects` router) or `CsrfProtect` (checks the CSRF token but doesn't require a prior session — used for anonymous/magic-link/ORCID-adjacent and share-link-join endpoints, where a session may not exist yet). `GET /auth/csrf` primes the `csrftoken` cookie for the SPA to read before its first POST.

**Rate limiting (`core/ratelimit.py`):** fixed-window counter on Django's default cache. Applied to anonymous login, magic-link request (per-email and per-IP), and share-link join. Documented limitation: LocMemCache is per-process, so this only holds under a single worker (fine for `runserver`/dev; a multi-worker prod deployment would need a shared cache backend — not in the current stack).

**CORS:** `django-cors-headers` added, since Phase 2's web app will call the API cross-port in dev. `CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS`/`FRONTEND_URL` are env-driven (defaults to `http://localhost:5173`).

**Acceptance check (Plan.md §9, Phase 1):**
- 29 Django tests (`accounts`, `projects`, `health`), all passing against real Postgres, both via `manage.py test` and via `pytest` (the CI path) — covers all three sign-in paths, magic-link single-use/expiry, stranger-blocked-without-share-link, anonymous-joins-and-is-scoped-to-that-project-only, role enforcement (viewer can't rename, only owner can create/revoke share links), CSRF enforcement (rejected without token, accepted with matched cookie+header).

## Phase 3 — Compilation service (sandboxed, dual compiler): DONE

Implements Plan.md §7 in full. This is the security-critical core of the project, so every control was verified against real behavior, not assumed — including one piece of research (latexmk's actual flag semantics) fetched from primary docs before writing any code, per CLAUDE.md's rule for exactly this situation.

**`apps/compile`** rewritten from the Phase 0 health stub into the actual orchestrator (`sandbox.py` + `main.py`). Verified against latexmk's real documentation before implementing, because memory was unreliable here and getting it wrong would silently leave shell-escape enabled: **latexmk has no native `-no-shell-escape` flag** — it must be baked directly into the `$pdflatex`/`$xelatex` command-override variable (`-e '$pdflatex=q/pdflatex -no-shell-escape .../'`), which is what `sandbox.py` does. Every §7 control is implemented and independently verified live (see below): no shell-escape, `--network none`, fresh container per job via the Docker Engine API (`cap_drop=["ALL"]`, `pids_limit`, `mem_limit`, `nano_cpus`, `read_only=True`, non-root `user=`, `no-new-privileges`), a wall-clock timeout enforced by `compile` itself (kills + removes the container on expiry, doesn't just abandon it), and a second independent path-safety check (`_safe_extract`) on top of the validation `projects/paths.py` already does, since untrusted-tar-becomes-filesystem-write is its own well-known vulnerability class (CVE-2007-4559-style) regardless of what already validated the logical paths upstream.

**Two real bugs found via testing, both fixed:**
1. **Docker-in-Docker sibling-container path bug.** `apps/compile` talks to the *host's* Docker daemon over the mounted socket to spawn job containers. `tempfile.mkdtemp()` was creating paths inside `compile`'s own private container filesystem — invisible to the host daemon, which resolves bind-mount sources against the *host* filesystem. Every job container therefore mounted an empty directory and reported "Could not find file 'main.tex'" despite the file genuinely existing on the `compile` side. Fixed by bind-mounting one real host directory (`./.compile-jobs`, via `${PWD}`) into `compile` at a fixed path, with `sandbox.py` writing job files there (visible to itself) while telling the *host* daemon the matching host-side path when specifying the sibling container's mounts. Documented in `sandbox.py` directly since it's exactly the kind of thing that's cheap to reintroduce.
2. My own first shell-escape test assertion was too weak to actually prove anything (an `or`-chain that was nearly always true). Replaced with real evidence: the compile log shows the exact `pdflatex -no-shell-escape ...` invocation and zero `runsystem` trace — TeX's documented behavior when shell-escape is disabled is to treat `\write18` as an inert `\write`, which is exactly what happened (compile succeeded normally, no command executed, no trace).

**`apps/api`**: new `ProjectSettings` (compiler, main_doc_path, central_bib_path — lazily created with defaults, matching Plan.md §8's schema which Phase 2 deliberately deferred) and `CompileRun` models. `projects/compile_api.py`: `GET`/`PATCH /projects/:id/settings`, `POST /projects/:id/compile` (materializes the project's current files from MinIO into an in-memory tar, dispatches to `compile`, stores the returned PDF/log/synctex back to MinIO, records a `CompileRun`), `GET /projects/:id/compile-runs[/​:id/pdf|/log]`. ## ORCID login 400 on user's production deployment (`freeleaf-api.deepmicroscopy.org`): RESOLVED

Root cause, found via the new logging (see below) within one round trip: `DisallowedHost` — the user's deployment routes the web app and the api to two *different* hostnames via their own reverse proxy (`freeleaf.deepmicroscopy.org` for the SPA, `freeleaf-api.deepmicroscopy.org` straight to the api), bypassing `apps/web/nginx.conf`'s same-origin proxying entirely. `DJANGO_ALLOWED_HOSTS` didn't include the api's own host, so Django rejected the request outright — not the session/cookie issue the vague original 400 suggested.

This is a valid, real-world deployment topology this repo's docs didn't cover — `.env.prod.example` and `docker-compose.prod.yml` only documented the same-origin-via-nginx setup, implicitly assuming `DJANGO_ALLOWED_HOSTS`/`FRONTEND_URL`/`CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS` are always the same one value. Fixed the docs, not just answered the user once:
- **`.env.prod.example`** now documents both topologies explicitly side by side: (A) same-origin via nginx — all four values identical; (B) split subdomains — `DJANGO_ALLOWED_HOSTS`/`ORCID_REDIRECT_URI` take the **api's** host, `FRONTEND_URL`/`CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS` take the **web app's** host (they're no longer the same value once the SPA's fetch() calls are genuinely cross-origin).
- **README**'s production section links to that and flags the specific failure signature (400 with nothing useful in the browser → check `docker compose -f docker-compose.prod.yml logs api` for `DisallowedHost` first).

The error-message-splitting and `LOGGING` config added while diagnosing (see previous revision of this entry) stay — they're generally useful, not just for this one bug, and are what actually surfaced the real cause in one round trip instead of several rounds of guessing.

**Separate follow-up while deploying:** `VITE_API_ORIGIN` (what the web app's JS uses to reach the api — needed for the split-subdomain topology above) was hardcoded to empty string in `docker-compose.prod.yml`'s build args, not configurable at all. ## Split-subdomain deployment: "CSRF token missing" creating projects — RESOLVED

Follow-up to the `DisallowedHost` fix above: once ORCID login worked (session cookie confirmed round-tripping correctly in the logs), `POST /api/projects` still failed with Django's `Forbidden (CSRF token missing.)`. Root cause: `CSRF_COOKIE_DOMAIN` defaults to host-only, so the csrftoken cookie set by the api's own host was invisible to `document.cookie` on the *web app's* different host — the frontend's CSRF middleware (`packages/shared/src/client.ts`) had nothing to echo back as `X-CSRFToken`. The session cookie didn't have this problem (subdomains are same-site, so the browser resends it automatically regardless of what JS can read) — only the CSRF double-submit pattern needs JS to actually read the cookie value, which is where host-only scoping bit.

Fix: new `CSRF_COOKIE_DOMAIN` setting (`config/settings.py`, `docker-compose.prod.yml`), set to the parent domain shared by both hosts (with a leading dot) — e.g. `.freeleaf.deepmicroscopy.org` for the user's actual `freeleaf.deepmicroscopy.org` / `api.freeleaf.deepmicroscopy.org` split. `.env.prod.example`'s topology-B section rewritten to use a sub-subdomain example (`api.freeleaf.example.com`) instead of a sibling one, since that's what the user ended up deploying and it makes the `CSRF_COOKIE_DOMAIN` value obviously-correct by construction. Attempted a local repro via fake `/etc/hosts` subdomains before the user settled on their final domain shape; abandoned that in favor of just shipping the fix once the topology was confirmed. **User confirmed working in production.**

## Phase 5 — Real-time collaboration: DONE

Yjs (§6, Node `collab` service — decision recorded in `docs/decisions/collab.md` back in Phase 0) now backs `.tex` editing: multi-cursor, live presence (including anonymous users), reconnect-after-drop convergence, and compiles reflecting merged state.

**Auth model.** `collab` has no DB/session access of its own by design (keeps Node confined to one service, per the ADR). `GET /projects/:id/files/:id/collab-token` (session-authed, reuses the existing membership/role check) mints a 60s HMAC-signed token (`core/collab_tokens.py` ↔ `apps/collab/src/token.ts` — same "`base64url(json).hex-hmac-sha256`" format on both ends, deliberately not JWT: one shared secret, one fixed shape, nothing to verify-against-live-docs since there's no library involved). The frontend passes it as a WS query param; `collab` verifies the signature+expiry locally, no round trip to `api` needed per connection. A second, separate surface — `/api/internal/collab/files/:id/content` (GET/PUT) — lets `collab` read/seed a room's initial content and periodically persist the merged plaintext back through api's existing MinIO-backed storage layer, protected by a shared-secret header (webhook-style trust model, not session auth, since there's no per-request user identity to check there).

**Wire protocol.** `y-websocket` v3 turned out to be client-only now (dropped its bundled server — a real, version-specific finding, not an assumption carried over from an older version). Hand-rolled the server side directly on `y-protocols/sync` + `y-protocols/awareness` + `lib0`, matching the browser client's exact outer message-type framing (verified by reading the installed client's own source, not from memory). Viewers get a read-only room: their sync step2/update messages are accepted-but-ignored server-side (in addition to the existing REST-layer role gate), so the WS path can't be used to bypass viewer restrictions.

**Persistence.** Each room seeds its `Y.Text` from the file's current stored content on first join (empty room → new `Room`), and flushes plaintext back every 4s if dirty, plus once more on the last client leaving (`apps/collab/src/room.ts`). No separate Yjs-binary-state storage — reconstructing a fresh `Y.Doc` from the last-known plaintext on next open was a deliberate simplification over persisting raw CRDT history, acceptable since content (not edit-history/undo-stack) is the thing that must never be lost across a fully-empty-room gap.

**Frontend.** `y-codemirror.next`'s `yCollab` extension binds directly to the shared `Y.Text`, giving sync + remote-cursor rendering (with name/color labels) for free. `CodeMirrorEditor` no longer does its own explicit save — content syncs continuously; the debounced "auto-compile on save" behavior from Phase 4 is now driven off `ydoc.on('update', ...)`, debounced 1500ms, only armed *after* the initial content sync (so just opening a file doesn't itself trigger a compile). A small presence row (colored initials, one per other connected user) sits next to a Live/Connecting/Reconnecting status pill.

**Three real bugs found and fixed during this pass** (all via live testing, not caught by `tsc`):
1. **Frontend forgot to regenerate the typed OpenAPI client** after adding the new `collab-token` endpoint — surfaced as a real prod build failure (`tsc` error on `api.GET(...)` not recognizing the new path) the user hit directly. Fixed by regenerating `packages/shared/src/generated.ts`, per this repo's own convention of generating types rather than hand-writing them (an in-flight hand-rolled `fetch()` workaround was reverted in favor of the generated client).
2. **`PDFDocumentProxy` doesn't have a `destroy()` method** (unrelated pre-existing Phase 4 code, caught mid-session while debugging something else) — it's on `PDFDocumentLoadingTask`. Fixed by holding onto the loading task itself.
3. **A genuine race in the hand-rolled collab server**: the browser's Yjs client sends its initial sync step1 synchronously on WS open, but the server only attached its `message` listener *after* `getOrCreateRoom`'s async content-fetch resolved — so that first message was silently dropped (Node's EventEmitter doesn't queue events for not-yet-attached listeners), the server never replied with step2, and the client never marked itself synced. Diagnosed with a minimal Node WS client script logging raw bytes on both ends (confirmed well-formed bytes were arriving; the gap was purely the missed first message) before fixing it: the message listener now attaches synchronously at connection time and buffers into a queue until the room is ready, then drains the queue before switching to live processing.
   - Also fixed along the way: a malformed reply where the server's sync-message handler always pre-wrote the outer message-type byte before checking whether there was anything to reply with, sending 1-byte non-replies for `update`/`step2` messages that a real client couldn't parse (`Unexpected end of array`).
   - And a `DJANGO_ALLOWED_HOSTS` gap identical in shape to the earlier ORCID production bug: `collab` calls `api` using the Docker-network hostname `api`, which wasn't in the allowed-hosts list, so every seed/persist call 400'd as `DisallowedHost`. Added `api` to both dev and prod defaults, documented in `.env.prod.example` alongside the existing topology notes (needed regardless of same-origin vs split-subdomain topology, since it's an internal Docker-network call either way).

**Live verification** (Playwright, two independent browser contexts — an invited editor via magic link, and a second anonymous "guest" joining the same share link, since anonymous-via-invite is exactly the real usage path per the auth model): both land on `main.tex`; each sees the other's presence dot; edits typed in one appear live in the other with no save/reload; dropping one client's connection and having the other keep editing, then reconnecting the dropped one, converges to all edits present with no data loss; clicking Recompile after all edits produces a successful compile. ## Track changes / diff markup (Reviewing mode): DONE

**Scope decision, made deliberately before writing code.** PLAN.md's literal description ("changes are tracked... render inline insertions with a green background/underline and deletions with a red background/strikethrough") most naturally maps to a live, per-keystroke, CRDT-attributed "suggestion layer" (Google-Docs-style Suggesting mode) — attributing every insertion/deletion to an author in real time on top of Yjs. **New `apps/web/src/components/workspace/trackChangesExtension.ts`:** a CodeMirror 6 `StateField<DecorationSet>` + `StateEffect`, computed by walking `diffLines(baselineText, currentText)` hunk by hunk while tracking a running position in the *current* document — unchanged/added hunks advance that position (real, editable live text, just marked green for insertions via `Decoration.mark`); removed hunks don't exist in the live document any more, so they're rendered as a `Decoration.widget` (a non-editable, injected `<span>`) showing the removed text with strikethrough, positioned exactly where it used to be.

**Real bug found and fixed during live verification: a CodeMirror re-entrancy crash.** First wiring recomputed decorations synchronously inside the `ydoc.on("update", ...)` listener — but that listener fires *while* `yCollab` is itself applying a Yjs update via a CodeMirror transaction (Yjs's own `update` event is emitted mid-`transact()`, itself nested inside a `view.update()` call already in progress), so dispatching another transaction synchronously from there hit CodeMirror's own re-entrancy guard: `"Calls to EditorView.update are not allowed while an update is in progress"` — a real console-level crash, not just a wrong-looking result. Fixed by deferring the recompute with a bare `setTimeout(..., 0)` to escape the current call stack before dispatching. ## Polishing-mode linter: DONE — Phase 8 complete

**Scope, per PLAN.md's own examples:** "detecting missing non-breaking spaces `~` before citations, orphaned headings, or unescaped symbols," plus aggressively surfacing LaTeX compile warnings/errors. New `apps/web/src/components/workspace/polishingLint.ts` — a pure-JS static scanner, explicitly documented as "best-effort, not a real LaTeX parser" (same framing as `log_parser.py`'s file-tracking), implementing:
- **Unescaped symbols** (`&`, `#`, `_`, `^`, and `%`) — with just enough context-tracking to stay useful without a full LaTeX parser: an `&` isn't flagged inside a tracked `tabular`/`align`-like environment; `_`/`^` aren't flagged inside a tracked math environment or inline `$...$` math (simple toggle, not `$$`); `#` isn't flagged when followed by a digit (macro parameters like `#1`). `$` itself is deliberately **not** checked — the compiler already validates balanced math delimiters far better than a static scanner could, and there's no reliable way to distinguish "meant currency" from "meant math" from source text alone.
- **Missing `~` before citations/references** — flags `\cite`/`\citep`/`\ref`/`\eqref`/`\Cref`/etc. immediately preceded by a plain space after a word character.
- **Orphaned headings** — a sectioning command immediately followed (skipping blank lines/`\label{}`) by another heading or `\end{document}`, meaning no body content follows. Explicitly documented as a document-*structure* proxy, not real page-layout analysis (which would need the rendered PDF, not just the source) — an honest scope boundary, not a claim of pixel-perfect pagination awareness.

**Real bug found and fixed during live verification: the `%` check was silently never firing.** First test input had "50%" appear *before* the deliberately-unescaped `&`/`_` later on the same line — and since an unescaped `%` correctly starts a real LaTeX comment (truncating the rest of the line from analysis, matching actual compiler behavior), those later characters were legitimately out of scope, correctly exposing that the scanner had **no check for `%` itself** at all — a real gap, since "forgot to escape %" (e.g. "50%" meant literally) is one of the most common LaTeX mistakes. Fixed by flagging `%` specifically when immediately preceded by a digit (a high-precision heuristic — genuine comments essentially never have a digit directly before the `%`), while still correctly treating it as the comment boundary for the rest of the line either way.

**Rendering:** reuses the exact same CodeMirror decoration pattern as Reviewing mode's track-changes markup (`polishingLintExtension.ts` — a `StateField`/`StateEffect` pair), rendering each finding as a wavy underline (`text-decoration: underline wavy`) with the message in a native `title` tooltip. Polishing mode's banner area becomes a scrollable findings list (`EditorTab.tsx`) merging lint findings with the most recent compile run's already-parsed errors/warnings (via a new `onRunChanged` callback on `CompilePane`) — each entry jumps to its line on click, reusing the same jump mechanism already built for comments and SyncTeX. Findings recompute live as you type (debounced via the same deferred-`setTimeout` pattern Reviewing mode needed, for the same CodeMirror re-entrancy reason).

### `\multicolumn`/`\multirow` support: DONE

Initially out of scope (blanket-rejected); extended once `\multicolumn` and `\multirow` (used separately, not combined on one cell) were asked for.

**Data model**: `TableGridModel.rows` changed from `string[][]` to `TableCell[][]`, a discriminated union — `text`, `multicolumn` (text, colspan, align, left/right border), `multirow` (text, rowspan, width), and `covered` (occupied by a span elsewhere). Every row still has exactly `columns.length` entries — covered slots fill the gaps — so the UI can always resolve (row, column) to one cell, span or not, without separately tracking raw-vs-visual column indices.

**Parsing**: the existing brace-depth-aware tokenizer already handled `\multicolumn{n}{spec}{text}`/`\multirow{n}{width}{text}` correctly as opaque cell text (their argument braces don't confuse `&`/`\\` detection) — the real work was reinterpreting each row's raw cells afterward: walking left to right, tracking which columns still owe an empty placeholder row for a `\multirow` in progress, and expanding a `\multicolumn{n}` into one real cell plus `n-1` synthetic `covered` cells. Explicitly rejects: a `\multicolumn` whose spec isn't a plain (optionally bordered) `l`/`c`/`r`; a `\multicolumn` overflowing the table width; a `\multirow` running past the last row; a non-empty cell where an empty `\multirow` placeholder was expected; a `\multicolumn` overlapping a column still pending a `\multirow` placeholder; and — the one deliberately out-of-scope combination — a cell nesting both (`\multicolumn{...}{...}{\multirow{...}}`).

**Serialization bug caught by the standalone verification script** (before any UI work): a `covered` cell was being treated as "just omit it," which is correct for a `\multicolumn`'s covered columns (no `&`-slot at all in the source) but wrong for a `\multirow`'s covered cells in later rows — those still need a real, empty `&`-slot; LaTeX has no way to omit it. Fixed by splitting `covered` into `{by: "colspan"}` (omit) vs `{by: "rowspan"}` (serialize as empty). Caught immediately by 2 of the round-trip checks (60 total) failing in the standalone `tsx` run before this ever reached the UI — the exact "verify the highest-risk logic standalone first" discipline paying off again.

**Editing UI**: a plain cell shows small merge controls (⇥ merge right into a `\multicolumn`, ⇓ merge down into a `\multirow`) whenever a mergeable plain neighbor exists — including extending an *existing* span further (mergeRight on a colspan-2 multicolumn merges with the cell after the span, growing it to colspan-3). A merged cell shows its own alignment/border toggles (or a width field for `\multirow`) plus a ✕ split control that reverts it to plain cells, keeping the original text in the top-left cell. Add/remove-row/column controls are disabled wherever any row/column has a cell that isn't plain `text` (`canRemoveColumn`/`canRemoveRow`) — rather than work out how to shrink or re-route a span through a removed row/column, the UI just asks the user to split first. Rendering is a straightforward `colSpan`/`rowSpan` walk: a `covered` cell (whichever kind of span put it there) simply isn't rendered at all, since the browser's own `colSpan`/`rowSpan` handling already accounts for it.

Verified live end-to-end (Playwright): typed a table with a pre-existing `\multirow{2}{*}{Group}` and `\multicolumn{2}{c}{Score}` → designer opened showing exactly 4 inputs (not 6 — covered cells correctly un-rendered), values `["Group","Score","A","B"]`, a real `colspan`/`rowspan` attribute present on the merged `<td>`s, a width input for the multirow → merged "A" and "B" via the ⇥ button → saved → source came back with the merge correctly serialized (`\multicolumn{2}{l}{A B}` plus the required empty placeholder cell before it) → reopened, split the `Group` multirow via ✕ → saved again → source correctly reverted to plain `Group` plus an empty placeholder cell, `Score`/`A B` spans untouched. ## Table Designer parser gaps: FIXED (real tables from a user bug report)

Fed the parser two real tables from an actual paper; both were rejected. Traced each rejection to a specific cause — two were genuine parser bugs (should never have been rejected), two were real scope gaps (booktabs was explicitly out of scope; omitted trailing cells weren't handled at all):

1. **`\\` wasn't brace-depth-aware.** `\hline`/`&` were already only treated as structural at brace depth 0, but the row-separator `\\` check wasn't — so a literal `\\` used for line-breaking *inside* a cell's own braces (`\thead{a\\b}`, `\shortstack{a\\b}`, etc.) was misread as an actual row break, corrupting the table. Fixed by adding the same `depth === 0` guard already used for `\hline`.
2. **Escaped `\&` wasn't distinguished from a real cell separator** — `H\&E` (a very ordinary way to write "H&E" staining in a histology paper, appeared twice in the second table) would split into two cells. Fixed by matching `\&` as a two-character literal before the bare-`&` check ever sees it.
3. **Trailing cells can be omitted in real LaTeX** — a row with fewer `&`-separated cells than the column spec just leaves the rest blank; LaTeX doesn't error. `interpretGrid` previously rejected any shortfall. Fixed by filling remaining columns with empty text (or an empty `\multirow` placeholder, if one was pending) once a row's raw cells run out — safe specifically *because* cells are consumed left-to-right, so running out can only mean a trailing omission, never a gap in the middle (which still produces a distinct, still-rejected "too many/wrong number of cells" case).
4. **Booktabs (`\toprule`/`\midrule`/`\bottomrule`) was explicitly out of scope.** Lifted: `horizontalBorders` changed from `boolean[]` to `(false | "hline" | "toprule" | "midrule" | "bottomrule")[]` — tracked per-slot (not just a single table-wide "style" flag) so a table round-trips back to the *exact* macro it came from even in the unusual case of mixing conventions. A new `defaultRuleKind` (the table's dominant style at parse time, `"hline"` for a table built from scratch) is what a newly-toggled-on border in the UI defaults to. The existing doubled-rule and "rule in the middle of a row" rejections apply uniformly across all four macros with no extra code, since they're now just different spellings of the same structural token.

Verified standalone via `tsx` (41 checks: regressions for the existing plain/multicolumn/multirow cases, one check per fix above, and both of the reporter's actual tables reduced to their essential constructs) before touching anything UI-related — same discipline as every other round on this parser. Then live-verified (Playwright) against the **exact, untrimmed** tables from the report: both now open in the designer instead of being rejected; table 1's cell grid correctly shows `\textbf{...}` header cells as raw LaTeX in their inputs and the `\multirow` spanning 10 rows with the omitted-trailing-cell last row intact; table 2 correctly shows the booktabs rules, all six `\thead{...\\...}` multi-line header cells, the escaped `H\&E` cells, and the `\cite{...}` in a data cell — edited one cell and saved, and the full document (both tables, untouched except the one edit) came back exactly right.

## Collab WebSocket reconnect after sleep/long disconnect: FIXED

User report: "when I wake the laptop from sleep, the reconnect does often not work." Traced to a real, fully-understood root cause rather than a vague browser quirk:

1. The collab WS token is deliberately short-lived — 60 seconds (`COLLAB_TOKEN_TTL_SECONDS` in `collab_api.py`), by design: "only needs to live long enough to open the WS connection."
2. `y-websocket`'s `WebsocketProvider` is constructed once with that token baked into its connection params, and **reuses the same params object for every automatic reconnect it ever makes** — it never refetches. Confirmed by reading the installed package's source (`node_modules/.../y-websocket/src/y-websocket.js`): `setupWS` connects to `provider.url`, and `url` is a getter that re-encodes `provider.params` on every read — meaning mutating `params.token` in place *is* enough to affect the next reconnect, but nothing in the library does that mutation on its own.
3. The collab server (`apps/collab/src/server.ts`) re-validates the token's expiry on every connection attempt, including reconnects, so any reconnect happening more than ~60s after the page loaded — sleep/wake being the most reliable way to trigger one, but really any connection drop — was permanently rejected. `y-websocket` has no way to know *why* the reconnect failed, so it just retried the same dead token forever with exponential backoff, stuck on "Reconnecting…".

Fix, in `CodeMirrorEditor.tsx`: a `setInterval` every 40 seconds (safely inside the 60s TTL) refetches a fresh token from the same `/collab-token` endpoint and writes it into `provider.params.token` — so whatever token is loaded when a reconnect actually happens is never more than ~40s old. On top of that, `visibilitychange`/`online` listeners immediately refresh the token and, if the socket isn't already connected, force a reconnect (`provider.disconnect()` + `provider.connect()`) right away rather than waiting on `y-websocket`'s own ~30s stale-connection detection to notice first — the common real trigger (waking a laptop, a backgrounded/throttled tab) gets a snappy recovery instead of an eventual one. All timers/listeners are torn down in the effect's cleanup.

Verified live (Playwright) two ways: (1) confirmed the periodic refresh actually fires — a fresh `/collab-token` request landed at ~40s, not just at initial connect; (2) the conclusive one — stayed connected for 75 real seconds (past the *original* token's 60s expiry), then forced a disconnect/reconnect cycle (toggling the browser context offline and back), and it **reconnected successfully** — which is only possible because the periodic refresh had already loaded a non-expired token into the provider before the reconnect was attempted. Without the fix this exact scenario is the one that was failing.

## Real bug found: comment highlights snapping back to a stale position while typing

The user's earlier "track-changes markings left-shift while typing" report turned out — once they could reproduce it again — to actually be about a **comment** anchor's highlight, not a suggestion decoration. Root cause, found by tracing the recompute trigger rather than guessing: `EditorTab.tsx`'s `commentAnchors` array (fed into `CodeMirrorEditor`) was rebuilt inline on *every render*, not memoized — and typing moves the cursor, which updates `cursorLine` state, which re-renders `EditorTab`, which happens on nearly every keystroke. Each of those fresh array references made `CodeMirrorEditor`'s `useEffect([commentAnchors, ...])` treat it as "the comment list changed" and fire a full decoration recompute — but that recompute rebuilt every anchor's position from its **stale, creation-time** `anchor_from`/`anchor_to` offsets stored on the `Comment` row, discarding whatever correct, live-drifted position CodeMirror's own `deco.map()` had already moved it to as the surrounding text changed. The comment highlight was — invisibly, almost every keystroke — snapping back to where it was when first created, which looked exactly like "left-shifting" or "not moving" while the real document kept growing around it.

Fixed at the source, not just the trigger: `computeCommentAnchorDecorations` now takes the field's own current live `DecorationSet` and prefers an existing comment's already-drifted position over its stored offsets — only a genuinely *new* comment (no live decoration yet) falls back to its stored `anchor_from`/`anchor_to`, since it hasn't had a chance to drift from anything. Also memoized `commentAnchors` in `EditorTab.tsx` against `comments` specifically, restoring the original intent ("recompute only when the comment list actually changes") as a secondary fix — belt and suspenders, since the first fix alone makes even a redundant recompute harmless.

Verified with a precise standalone reproduction (real `EditorState` + the actual `commentAnchorsField`, no mocking): created an anchor, typed text before it to drift its live position, then recomputed using the exact same stale stored offsets that used to cause the bug — confirmed the drifted position survives instead of snapping back, confirmed a genuinely new comment still uses its own stored offsets, and confirmed the old (pre-fix) behavior is still reproducible by omitting the new parameter, so the regression itself stays covered.

## Real bug: `\setmainfont{Arial}` (and other Office fonts) failing under XeLaTeX

User report: `Package fontspec Error: The font "Arial" cannot be found.` This isn't a "missing TeX package" gap the way `biblatex-ieee` was (texlive-full fix, earlier round) — Arial, Times New Roman, Calibri, and Cambria are proprietary Microsoft fonts that Microsoft doesn't license for redistribution, so no TeX Live distribution, however complete, will ever include them. Fixing this needed real font substitution.

Confirmed by reproducing it directly against the sandbox image (`docker run ... xelatex \setmainfont{Arial}`) rather than guessing. The metric-compatible open substitutes were already present as transitive dependencies of `texlive-full` (Arimo/Tinos/Cousine via `fonts-croscore`, Carlito/Caladea via `fonts-crosextra-*`) — `fc-match Arial` already resolved to Arimo correctly. The surprising part: **fontconfig-level substitution wasn't enough.** Tried, in order, and confirmed each still failed by actually recompiling: an `<alias>`/`<accept>` rule, a `target="pattern"` `<match>` (query-time substitution), and a `target="scan"` `<match>` with both `append` and `prepend` (font-cache-time renaming) — every one of these left `fc-match`/`fc-list` correctly showing "Arial" as a valid name for the Arimo files, yet XeTeX's own internal font manager (`fontspec`/`XeTeXFontMgr_FC`) still failed to find it, evidently because it indexes fonts by their file's own *primary* declared name and doesn't consult whichever fontconfig alias/rewrite layer produced that display output.

Kept as its own Docker build stage (`FROM debian:bookworm-slim AS fontgen`) so the ~7GB final image doesn't need `fonttools` and its fairly heavy `scipy`/`pythran` dependency chain at all — only the four generated `.ttf` files per family get copied into the real image via `COPY --from=fontgen`.

Verified live by actually compiling `\setmainfont{...}` for all five covered names (Arial, Times New Roman, Courier New, Calibri, Cambria) against the rebuilt image and confirming both a clean log (no fontspec errors) and real extractable PDF text (`pdftotext`), not just "compiled without crashing."

## LaTeX syntax highlighting (planned via `/plan`, then implemented)

Confirmed live (zero `<span>`s inside any `.cm-line`) that the editor had **no syntax highlighting at all** — `CodeMirrorEditor.tsx` imported `StreamLanguage.define(stex)` from `@codemirror/legacy-modes/mode/stex` but never added a `syntaxHighlighting(...)` extension, so every token rendered as identical plain text.

## Real bug: compiles with a recoverable LaTeX error produced no PDF at all

User: "Sometimes the latex compile does not complete, although it could." Root cause, found in `apps/compile/sandbox.py`: both the `pdflatex`/`xelatex` engine invocation (baked into `ENGINE_COMMANDS`) and the `latexmk` command line passed `-halt-on-error` — which stops the engine dead at the *first* error in the document. For pdfTeX-family engines that means no PDF is written at all, even for a document with one trivial, fully-recoverable issue (an undefined `\ref`, a typo'd command) that `-interaction=nonstopmode` alone would have skipped past, finishing the rest of the document into a perfectly usable PDF — matching how Overleaf and most real LaTeX tooling actually behaves.

Fixed by removing `-halt-on-error` from both places (keeping `-interaction=nonstopmode`, which is the actual requirement in this non-interactive sandbox — it guarantees the batch run never blocks waiting for input, a distinct concern from halting-on-error) and adding latexmk's `-f` ("force continued processing past errors," verified against the real installed binary via `latexmk --help` before using it, not assumed from memory) so latexmk's own multi-pass recipe — reruns for cross-references, bibtex/biber — also keeps going rather than aborting the whole build after one pass reports an error.

This changed the shape of a case `sandbox.py`'s `status` field wasn't handling well: with errors no longer halting the engine, a nonzero `latexmk`/engine exit code paired with a real, valid PDF having still landed became the *common* case for any document with a recoverable error, not a rare edge case. `status` previously keyed directly off the raw exit code (`"success" if exit_code == 0 else "failed"`) with only a one-way downgrade if a PDF was unexpectedly missing — so a document that fully compiled despite a minor error would get mislabeled "Compile failed" even though a PDF was sitting right there (the frontend's `CompilePane.tsx` already renders whenever `has_pdf` is true, independent of `status`, so the PDF itself would have still shown — just under a contradictory-looking red "failed" label). Flipped the logic: `status` now starts `"success"` and is only downgraded to `"failed"` if no PDF actually landed — "did we get something to show you" instead of "was every tool invocation 100% clean." Parsed errors/warnings (`log_parser.py`) are completely unaffected either way and still surface in the Diagnostics list regardless of status.

Verified live end-to-end via the real full compile pipeline (rebuilt the `compile` service image — it has no dev bind-mount, so a plain restart wouldn't have picked up the change): a document with `\thiscommanddoesnotexist` in its body now shows **"Compiled successfully,"** renders a real PDF, and still lists "Undefined control sequence" in the diagnostics panel — previously this exact case produced no PDF at all. Confirmed the negative case is untouched: a genuinely fatal document (no `\documentclass`, an unterminated group straight to EOF) still correctly reports "Compile failed" with no PDF rendered.

## Real bug: compile stuck indefinitely on "Compiling…" in production

Follow-up to the fix above. User's real-world document (a 35MB zip with two undownsampled ~17MB PNG figures) needed the compiler switched to `xelatex` (its class requires `fontspec`, which needs XeTeX/LuaTeX — confirmed by importing the actual zip and reading the real compile log rather than guessing), but even after that fix the user reported the spinner never resolving, specifically in production, and specifically "indefinitely" rather than settling into a definitive error after a minute or so. Reproducing the same zip on this local dev stack didn't hang — it completed in ~40s, close to but under the sandbox's 60s wall-clock cap, which was the first clue: this document's compile time is genuinely long (large raw images), and a production environment need only be a bit slower or more loaded to push past whatever ceiling was actually acting as the limit.

Two independent, compounding bugs found, both real:

1. **`apps/web/nginx.conf`'s `/api/` location block had no `proxy_read_timeout` override**, so it silently inherited nginx's built-in default of 60 seconds — shorter than the compile pipeline's own worst-case ceiling (the sandbox's 60s wall clock *plus* MinIO materialization, sibling-container spin-up, and the API's own 90s upstream timeout waiting for the compile service). A compile that would have finished fine given a few more seconds instead gets cut off by nginx returning a 504 out from under it. (`/collab/`'s location block already had `proxy_read_timeout 3600s` for its WebSocket connections — `/api/` was simply missed.) Fixed: added `proxy_read_timeout 120s` to `/api/` too.
2. **`CompilePane.tsx`'s `compile()` had no `try`/`catch`/`finally`** around the compile request. `openapi-fetch` doesn't throw for a well-formed JSON error response, but nginx's default 504 error page is HTML, not JSON — trying to parse it throws, and since nothing was catching that, execution jumped straight out of `compile()` without ever reaching `setCompiling(false)`. The UI was left permanently showing "Compiling…", with no toast, no re-enabled button, and no way to recover short of a full page reload — this is what actually explains "indefinitely," independent of and on top of bug 1 (bug 1 explains *why* a 504 happened at all for this particular document; bug 2 explains why the frontend never recovered from it, and would still not recover from a 504 caused by anything else — a MinIO blip, a compile-service restart, a genuinely oversized document — even after bug 1 is fixed). Fixed with a `try`/`catch`/`finally`: the `finally` block unconditionally resets `compiling`/`inFlightRef`, and the `catch` shows a toast ("Lost the connection while compiling — please try again.") so the user gets a clear, actionable signal instead of silence.

Verified live: (1) confirmed via `nginx -t` inside a real `Dockerfile.prod` build that the new `proxy_read_timeout` line is syntactically valid; (2) reproduced the exact production failure mode with Playwright request interception — routed the compile POST to return a real HTTP 504 with an HTML body (not mocked as a JSON error, the actual shape nginx sends) — and confirmed the fixed frontend shows the toast within one render, clears the spinner, and re-enables Recompile, instead of hanging forever as it did before the fix.

## Follow-up: the real production failure mode was gunicorn's own 30s default timeout, not nginx's

User's actual production console after the fix above: a **500** (not 504) on the compile request, reported by the browser as a CORS failure (`Origin ... is not allowed by Access-Control-Allow-Origin. Status code: 500`) — a classic misdirection: CORS headers are attached by Django itself, so any failure that kills the request *before* Django finishes (rather than Django deliberately returning an error response) produces a response with no CORS headers at all, which the browser reports as a CORS problem even though the real cause is unrelated to CORS policy.

The browser console also revealed something the previous round's fix couldn't have addressed: the user's production deployment routes `api.freeleaf.deepmicroscopy.org` through their **own separate reverse proxy**, not through `apps/web/nginx.conf` at all (the split-subdomain topology documented back in the `DisallowedHost`/`CSRF_COOKIE_DOMAIN` fixes) — so the previous round's `proxy_read_timeout` fix, while still correct for the same-origin topology, was never going to touch this deployment's actual bottleneck.

That bottleneck: **gunicorn's own `--timeout` was never set anywhere** (`docker-compose.prod.yml`, `apps/api/Dockerfile.prod`), so it silently used gunicorn's built-in default of **30 seconds** — a worker still handling a request past that gets SIGKILL'd and restarted, regardless of what any proxy in front of it allows. Since this user's real documents can legitimately take 40–90+ seconds to compile (large, undownsampled images), gunicorn was killing the Django worker itself well before the compile pipeline's own generous internal timeouts (sandbox 60s, `compile_api.py`'s own 90s upstream wait) ever got a chance to matter — the dropped connection is what their own reverse proxy then turned into the 500 seen in the console.

Fixed: `--timeout 100` added to gunicorn in both `docker-compose.prod.yml` (the one actually in effect — its `command:` overrides the image's baked-in `CMD`) and `apps/api/Dockerfile.prod`'s `CMD` (kept in sync for anyone running the image directly). 100s sits deliberately just above `compile_api.py`'s own 90s ceiling, so Django gets to finish and return a clean, well-formed response — success, failure, or a clean "timeout" status — before gunicorn would give up on it, and `apps/web/nginx.conf`'s 120s (previous round) has headroom past *that*. Documented the whole chain in a comment at the gunicorn command itself, and added a paragraph to the README's split-subdomain section flagging that **a user's own external reverse proxy needs its own timeout raised to match** — that's infrastructure outside this repo, and the previous round's nginx fix couldn't have helped this deployment shape at all without also knowing this.

Verified gunicorn actually accepts `--timeout 100` and boots cleanly (built the real `Dockerfile.prod` image, ran it, confirmed 3 workers started with no startup errors) and that `docker-compose.prod.yml` still parses as valid YAML after the edit.

## Real bug: the PDF "Download" button only opened the PDF, never actually downloaded it

Root cause: `CompilePane.tsx`'s Download control was a plain `<a href=".../pdf" download="...">`. The `download` attribute is only a reliable *force-download* hint for same-origin URLs — for a cross-origin target (dev's separate `:5173`/`:8000` ports, or — the user's actual case — a split-subdomain production deployment with the api on its own host) browsers just navigate to/render the resource instead, silently ignoring the attribute entirely. An earlier round of this project had already noticed a *related* symptom (the suggested filename falling back to generic in dev) but wrongly concluded it was cosmetic and would "resolve itself" in production via same-origin nginx — that assumption doesn't hold for this user's actual split-subdomain deployment, where the same root cause fully breaks the download, not just the filename.

Fixed by not relying on the `download` attribute against a cross-origin URL at all: `handleDownload` now `fetch()`es the PDF itself (`credentials: "include"`, so the session cookie still rides along cross-origin same as before), reads it as a `Blob`, and downloads it via a `blob:` object URL — which is always same-origin to the page regardless of where the bytes actually came from, so the force-download behavior is reliable everywhere. The `<a>` became a `<button>` (no longer a real navigable link), reusing the exact same `.downloadLink` CSS class — already fully self-contained (background/border/color/padding), so no new styling needed, plus a `downloading` state for a brief "Downloading…" label during the fetch.

Verified live in the exact failure scenario (Playwright, `localhost:5173` → `localhost:8000`, mirroring the split-subdomain topology): a real browser "download" event now fires (previously the click would have just navigated), with the correct project-name-based filename (not the generic fallback) and valid, complete PDF bytes.

## Pattern-matched LaTeX fix-it assistant (not LLM-based)

User request: a purely pattern-matched assistant (explicitly not LLM-based) that recognizes common, mechanically-fixable compile problems and offers a one-click/one-dialog fix — add a missing package, locate/upload a missing file, resolve a duplicate label — plus "a cute baby penguin trying to help" illustrated in every fix dialog.

**Grounded the rule list in real behavior before designing anything** (planned via `/plan`): compiled 6 minimal documents against the real sandbox to confirm exact log wording. Two real findings that shaped the design: (1) `! No schema change. Two new real-log fixtures (`UNDEF_TOPRULE_LOG`, `UNDEF_MULTIROW_LOG`) added to `test_log_parser.py`; two existing tests elsewhere (`test_log_parser.py`, `test_compile.py`) updated for the new message format. ## Real bug: no suggested fixes (and no diagnostics at all) for a document using `\bibliography`/bibtex

User report: a project that compiles fine but never shows the fix-it assistant's suggestions, despite genuinely having a couple of fixable problems (`\multirow`/`\lipsum` used without their packages). Reproduced with the user's actual zip, compiled through the real sandbox, and found something worse than a fix-it-specific bug: `run.errors`/`run.warnings` came back **completely empty** for this document — the pre-existing Diagnostics list (not just the new Suggested Fixes) was silently showing nothing at all, even though the raw log clearly contained the undefined-command errors and several undefined-citation warnings.

Root cause in `log_parser.py`'s `_last_engine_run`: its `_RUN_START_RE` (`^This is \S*TeX,`), meant to isolate latexmk's *final* pdflatex/xelatex/lualatex rerun from all the concatenated reruns in the captured log, also matches **BibTeX's own** "This is BibTeX, Version ..." banner (Bib + TeX). For a `\bibliography`-using document (classic bibtex, not biblatex/biber), latexmk's real rule order can legitimately end on a bibtex invocation (pdflatex → bibtex → pdflatex → bibtex again) — so bibtex's banner was, correctly by the regex's own logic but wrongly in intent, the *last* "This is ...TeX," line in the whole log. Every diagnostic from the actual final pdflatex run was silently discarded, entirely unreported, no error surfaced anywhere. Fixed by anchoring the regex to the three real engines by name (`^This is (?:pdf|Xe|Lua)TeX,`) instead of any tool whose name happens to end in "TeX".

New regression fixture (`TRAILING_BIBTEX_RUN_LOG`, `test_log_parser.py`) reproduces the exact shape (pdflatex run with a real error → bibtex → pdflatex rerun with the same error + warnings → a second bibtex run as the log's actual last line) and asserts the second pdflatex run's diagnostics are correctly recovered, not discarded. Full suite: 265/265 passing. Verified against the user's real zip end-to-end: re-imported it, recompiled, and confirmed the API response now correctly returns the `\multirow`/`\lipsum` errors and all the undefined-citation warnings (previously all silently empty) — and the fix-it assistant now correctly shows both "add package" suggestions.

## Fix-it assistant: auto-open a suggested fix's dialog after compiling

Follow-up: seeing the suggestions only in the (easy-to-miss) diagnostics list wasn't enough — the user wanted the dialog to open by itself when a compile finds a fixable problem, configurable, defaulting to on.

**Scope decision.** Made this a personal, client-side preference (`localStorage`, keyed `freeleaf.autoOpenFixes.{projectId}`, defaulting to `true` when unset) rather than a shared `ProjectSettings` field — same reasoning as the editing-mode selector's own local-only split (Phase 8): whether an auto-popping dialog is helpful or intrusive is an individual's tolerance, not something that should be forced the same for every collaborator in a shared session. A small checkbox ("Open automatically after compiling") sits inline in `CompilePane.tsx`'s new "Suggested fixes" heading, appearing exactly when it's relevant rather than cluttering the toolbar at all times.

**Real UX problem caught before shipping, not after:** a naive "auto-open on every compile" would have fired on the *silent, debounced auto-compile that already runs after every pause in typing* (Phase 5's `ydoc.on('update', ...)` → `scheduleAutoCompile`), popping a modal and stealing focus from the editor mid-keystroke every ~1.5s while an unresolved error existed — exactly the "intrusive" failure mode the preference exists to let people avoid, and arguably worse than not having the feature at all. Fixed by threading an `explicit: boolean` through `compile()`: only `triggerCompile()` (the Recompile button, Cmd+S, and the fix dialogs' own post-apply recompiles) passes `explicit=true` and is eligible to auto-open; `scheduleAutoCompile()`'s debounced timer always passes `false`. The existing in-flight-request coalescing (`rerunRequestedRef`) also tracks whether the *deferred* rerun was itself requested explicitly, so a Recompile click that lands while an auto-compile is still running doesn't lose its explicitness.

Only the *first* matched candidate auto-opens (never a cascade of stacked dialogs); any remaining ones stay available as ordinary "Fix" buttons in the list. Verified live (Playwright, four scenarios): typing alone (silent auto-compile) never opens a dialog; clicking Recompile with the default preference does; unchecking the toggle and recompiling again does not; reloading the page and recompiling confirms the unchecked preference persisted.
