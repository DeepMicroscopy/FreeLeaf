## Phase 1 ‚Äî Identity & access: DONE

Two new Django apps in `apps/api`: `accounts` (User, MagicLink) and `projects` (Project, Membership, ShareLink) ‚Äî schema per Plan.md ¬ß8. No passwords anywhere; `django.contrib.auth` stays reserved for Django admin staff logins only. Session-based app identity lives in `request.fl_user`, set by `accounts.middleware.CurrentUserMiddleware` from a `fl_user_id` session key (`core/session.py`).

**Sign-in paths (`accounts/api.py`):**
- **ORCID OAuth2** (`accounts/orcid.py`) ‚Äî verified against ORCID's live API tutorial before building (2026-07-10). One deviation from Plan.md's literal wording: scope `openid` instead of `/authenticate`, per ORCID's own current guidance that they're equivalent and `/authenticate` is superseded ‚Äî recorded in `docs/decisions/orcid-scope.md`. Sandbox by default (`ORCID_ENV=sandbox`). State-parameter CSRF protection on the OAuth redirect. Trusts the `orcid`/`name` fields ORCID's token endpoint returns directly (documented behavior, no extra API call).
- **Magic link** (`accounts/magic_link.py`) ‚Äî signed random token (sha256-hashed at rest, raw token only ever emailed), 15 min TTL, single-use, delivered via Mailpit in dev. Email normalized to lowercase to avoid duplicate accounts.
- **Anonymous** ‚Äî `POST /auth/anonymous`, optional `display_name`.

**Authorization (`projects/`):** Project/Membership/ShareLink CRUD. `projects.authz.get_authorized_project` returns 404 (not 403) for both "doesn't exist" and "no membership," so unauthorized callers can't distinguish the two. Anonymous users get access **only** by hitting `POST /share-links/{token}/join`, which creates their `User` row and `Membership` together ‚Äî there's no other path to a Membership for an anonymous kind user. Share-link tokens are sha256-hashed at rest like magic-link tokens (spec names the field `token`; storing it hashed is a deliberate hardening beyond the literal spec, not a behavior change).

**CSRF/sessions:** Django's `CsrfViewMiddleware` doesn't actually protect django-ninja views (ninja marks all its views `csrf_exempt` at the Django level by design, and re-implements the check itself inside its cookie-auth classes) ‚Äî found this the hard way via a failing test, not by inference. So every state-changing endpoint explicitly carries a cookie-based auth class: `SessionAuth` (requires an existing session ‚Äî used for the `projects` router) or `CsrfProtect` (checks the CSRF token but doesn't require a prior session ‚Äî used for anonymous/magic-link/ORCID-adjacent and share-link-join endpoints, where a session may not exist yet). `GET /auth/csrf` primes the `csrftoken` cookie for the SPA to read before its first POST.

**Rate limiting (`core/ratelimit.py`):** fixed-window counter on Django's default cache. Applied to anonymous login, magic-link request (per-email and per-IP), and share-link join. Documented limitation: LocMemCache is per-process, so this only holds under a single worker (fine for `runserver`/dev; a multi-worker prod deployment would need a shared cache backend ‚Äî not in the current stack).

**CORS:** `django-cors-headers` added, since Phase 2's web app will call the API cross-port in dev. `CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS`/`FRONTEND_URL` are env-driven (defaults to `http://localhost:5173`).

**Two real bugs found via testing, both fixed:**
1. **Docker-in-Docker sibling-container path bug.** `apps/compile` talks to the *host's* Docker daemon over the mounted socket to spawn job containers. `tempfile.mkdtemp()` was creating paths inside `compile`'s own private container filesystem ‚Äî invisible to the host daemon, which resolves bind-mount sources against the *host* filesystem. Every job container therefore mounted an empty directory and reported "Could not find file 'main.tex'" despite the file genuinely existing on the `compile` side. Fixed by bind-mounting one real host directory (`./.compile-jobs`, via `${PWD}`) into `compile` at a fixed path, with `sandbox.py` writing job files there (visible to itself) while telling the *host* daemon the matching host-side path when specifying the sibling container's mounts. Documented in `sandbox.py` directly since it's exactly the kind of thing that's cheap to reintroduce.
2. My own first shell-escape test assertion was too weak to actually prove anything (an `or`-chain that was nearly always true). Replaced with real evidence: the compile log shows the exact `pdflatex -no-shell-escape ...` invocation and zero `runsystem` trace ‚Äî TeX's documented behavior when shell-escape is disabled is to treat `\write18` as an inert `\write`, which is exactly what happened (compile succeeded normally, no command executed, no trace).

## ORCID login 400 on user's production deployment (`freeleaf-api.deepmicroscopy.org`): RESOLVED

Root cause, found via the new logging (see below) within one round trip: `DisallowedHost` ‚Äî the user's deployment routes the web app and the api to two *different* hostnames via their own reverse proxy (`freeleaf.deepmicroscopy.org` for the SPA, `freeleaf-api.deepmicroscopy.org` straight to the api), bypassing `apps/web/nginx.conf`'s same-origin proxying entirely. `DJANGO_ALLOWED_HOSTS` didn't include the api's own host, so Django rejected the request outright ‚Äî not the session/cookie issue the vague original 400 suggested.

This is a valid, real-world deployment topology this repo's docs didn't cover ‚Äî `.env.prod.example` and `docker-compose.prod.yml` only documented the same-origin-via-nginx setup, implicitly assuming `DJANGO_ALLOWED_HOSTS`/`FRONTEND_URL`/`CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS` are always the same one value. Fixed the docs, not just answered the user once:

## Split-subdomain deployment: "CSRF token missing" creating projects ‚Äî RESOLVED

Follow-up to the `DisallowedHost` fix above: once ORCID login worked (session cookie confirmed round-tripping correctly in the logs), `POST /api/projects` still failed with Django's `Forbidden (CSRF token missing.)`. Root cause: `CSRF_COOKIE_DOMAIN` defaults to host-only, so the csrftoken cookie set by the api's own host was invisible to `document.cookie` on the *web app's* different host ‚Äî the frontend's CSRF middleware (`packages/shared/src/client.ts`) had nothing to echo back as `X-CSRFToken`. The session cookie didn't have this problem (subdomains are same-site, so the browser resends it automatically regardless of what JS can read) ‚Äî only the CSRF double-submit pattern needs JS to actually read the cookie value, which is where host-only scoping bit.

Fix: new `CSRF_COOKIE_DOMAIN` setting (`config/settings.py`, `docker-compose.prod.yml`), set to the parent domain shared by both hosts (with a leading dot) ‚Äî e.g. `.freeleaf.deepmicroscopy.org` for the user's actual `freeleaf.deepmicroscopy.org` / `api.freeleaf.deepmicroscopy.org` split. `.env.prod.example`'s topology-B section rewritten to use a sub-subdomain example (`api.freeleaf.example.com`) instead of a sibling one, since that's what the user ended up deploying and it makes the `CSRF_COOKIE_DOMAIN` value obviously-correct by construction. Attempted a local repro via fake `/etc/hosts` subdomains before the user settled on their final domain shape; abandoned that in favor of just shipping the fix once the topology was confirmed. **User confirmed working in production.**

## Phase 5 ‚Äî Real-time collaboration: DONE

Yjs (¬ß6, Node `collab` service ‚Äî decision recorded in `docs/decisions/collab.md` back in Phase 0) now backs `.tex` editing: multi-cursor, live presence (including anonymous users), reconnect-after-drop convergence, and compiles reflecting merged state.

**Auth model.** `collab` has no DB/session access of its own by design (keeps Node confined to one service, per the ADR). `GET /projects/:id/files/:id/collab-token` (session-authed, reuses the existing membership/role check) mints a 60s HMAC-signed token (`core/collab_tokens.py` ‚Üî `apps/collab/src/token.ts` ‚Äî same "`base64url(json).hex-hmac-sha256`" format on both ends, deliberately not JWT: one shared secret, one fixed shape, nothing to verify-against-live-docs since there's no library involved). The frontend passes it as a WS query param; `collab` verifies the signature+expiry locally, no round trip to `api` needed per connection. A second, separate surface ‚Äî `/api/internal/collab/files/:id/content` (GET/PUT) ‚Äî lets `collab` read/seed a room's initial content and periodically persist the merged plaintext back through api's existing MinIO-backed storage layer, protected by a shared-secret header (webhook-style trust model, not session auth, since there's no per-request user identity to check there).

**Wire protocol.** `y-websocket` v3 turned out to be client-only now (dropped its bundled server ‚Äî a real, version-specific finding, not an assumption carried over from an older version). Hand-rolled the server side directly on `y-protocols/sync` + `y-protocols/awareness` + `lib0`, matching the browser client's exact outer message-type framing (verified by reading the installed client's own source, not from memory). Viewers get a read-only room: their sync step2/update messages are accepted-but-ignored server-side (in addition to the existing REST-layer role gate), so the WS path can't be used to bypass viewer restrictions.

**Persistence.** Each room seeds its `Y.Text` from the file's current stored content on first join (empty room ‚Üí new `Room`), and flushes plaintext back every 4s if dirty, plus once more on the last client leaving (`apps/collab/src/room.ts`). No separate Yjs-binary-state storage ‚Äî reconstructing a fresh `Y.Doc` from the last-known plaintext on next open was a deliberate simplification over persisting raw CRDT history, acceptable since content (not edit-history/undo-stack) is the thing that must never be lost across a fully-empty-room gap.

**Frontend.** `y-codemirror.next`'s `yCollab` extension binds directly to the shared `Y.Text`, giving sync + remote-cursor rendering (with name/color labels) for free. `CodeMirrorEditor` no longer does its own explicit save ‚Äî content syncs continuously; the debounced "auto-compile on save" behavior from Phase 4 is now driven off `ydoc.on('update', ...)`, debounced 1500ms, only armed *after* the initial content sync (so just opening a file doesn't itself trigger a compile). A small presence row (colored initials, one per other connected user) sits next to a Live/Connecting/Reconnecting status pill.

**Three real bugs found and fixed during this pass** (all via live testing, not caught by `tsc`):
1. **Frontend forgot to regenerate the typed OpenAPI client** after adding the new `collab-token` endpoint ‚Äî surfaced as a real prod build failure (`tsc` error on `api.GET(...)` not recognizing the new path) the user hit directly. Fixed by regenerating `packages/shared/src/generated.ts`, per this repo's own convention of generating types rather than hand-writing them (an in-flight hand-rolled `fetch()` workaround was reverted in favor of the generated client).
2. **`PDFDocumentProxy` doesn't have a `destroy()` method** (unrelated pre-existing Phase 4 code, caught mid-session while debugging something else) ‚Äî it's on `PDFDocumentLoadingTask`. Fixed by holding onto the loading task itself.
3. **A genuine race in the hand-rolled collab server**: the browser's Yjs client sends its initial sync step1 synchronously on WS open, but the server only attached its `message` listener *after* `getOrCreateRoom`'s async content-fetch resolved ‚Äî so that first message was silently dropped (Node's EventEmitter doesn't queue events for not-yet-attached listeners), the server never replied with step2, and the client never marked itself synced. Diagnosed with a minimal Node WS client script logging raw bytes on both ends (confirmed well-formed bytes were arriving; the gap was purely the missed first message) before fixing it: the message listener now attaches synchronously at connection time and buffers into a queue until the room is ready, then drains the queue before switching to live processing.
   - Also fixed along the way: a malformed reply where the server's sync-message handler always pre-wrote the outer message-type byte before checking whether there was anything to reply with, sending 1-byte non-replies for `update`/`step2` messages that a real client couldn't parse (`Unexpected end of array`).
   - And a `DJANGO_ALLOWED_HOSTS` gap identical in shape to the earlier ORCID production bug: `collab` calls `api` using the Docker-network hostname `api`, which wasn't in the allowed-hosts list, so every seed/persist call 400'd as `DisallowedHost`. Added `api` to both dev and prod defaults, documented in `.env.prod.example` alongside the existing topology notes (needed regardless of same-origin vs split-subdomain topology, since it's an internal Docker-network call either way).

**Live verification** (Playwright, two independent browser contexts ‚Äî an invited editor via magic link, and a second anonymous "guest" joining the same share link, since anonymous-via-invite is exactly the real usage path per the auth model): both land on `main.tex`; each sees the other's presence dot; edits typed in one appear live in the other with no save/reload; dropping one client's connection and having the other keep editing, then reconnecting the dropped one, converges to all edits present with no data loss; clicking Recompile after all edits produces a successful compile. Screenshots + script: `pw/drive_phase5.mjs`.

**Known limitations, called out rather than silently deferred:** no automated test coverage for `apps/collab` itself (verified live only ‚Äî Node has no test setup in this repo yet); if multiple clients are typing when the debounce settles simultaneously, more than one may independently trigger a compile (wasteful, not incorrect); a maliciously crafted step2 message could theoretically still slip a viewer's edit through the WS-layer read-only gate (the real enforcement is the REST API's role check, this is defense-in-depth only, noted in `server.ts`).
