## Phase 1 — Identity & access: DONE

Two new Django apps in `apps/api`: `accounts` (User, MagicLink) and `projects` (Project, Membership, ShareLink) — schema per Plan.md §8. No passwords anywhere; `django.contrib.auth` stays reserved for Django admin staff logins only. Session-based app identity lives in `request.fl_user`, set by `accounts.middleware.CurrentUserMiddleware` from a `fl_user_id` session key (`core/session.py`).

**Sign-in paths (`accounts/api.py`):**
- **ORCID OAuth2** (`accounts/orcid.py`) — verified against ORCID's live API tutorial before building (2026-07-10). One deviation from Plan.md's literal wording: scope `openid` instead of `/authenticate`, per ORCID's own current guidance that they're equivalent and `/authenticate` is superseded — recorded in `docs/decisions/orcid-scope.md`. Sandbox by default (`ORCID_ENV=sandbox`). State-parameter CSRF protection on the OAuth redirect. Trusts the `orcid`/`name` fields ORCID's token endpoint returns directly (documented behavior, no extra API call).
- **Magic link** (`accounts/magic_link.py`) — signed random token (sha256-hashed at rest, raw token only ever emailed), 15 min TTL, single-use, delivered via Mailpit in dev. Email normalized to lowercase to avoid duplicate accounts.
- **Anonymous** — `POST /auth/anonymous`, optional `display_name`.

**Authorization (`projects/`):** Project/Membership/ShareLink CRUD. `projects.authz.get_authorized_project` returns 404 (not 403) for both "doesn't exist" and "no membership," so unauthorized callers can't distinguish the two. Anonymous users get access **only** by hitting `POST /share-links/{token}/join`, which creates their `User` row and `Membership` together — there's no other path to a Membership for an anonymous kind user. Share-link tokens are sha256-hashed at rest like magic-link tokens (spec names the field `token`; storing it hashed is a deliberate hardening beyond the literal spec, not a behavior change).

**CSRF/sessions:** Django's `CsrfViewMiddleware` doesn't actually protect django-ninja views (ninja marks all its views `csrf_exempt` at the Django level by design, and re-implements the check itself inside its cookie-auth classes) — found this the hard way via a failing test, not by inference. So every state-changing endpoint explicitly carries a cookie-based auth class: `SessionAuth` (requires an existing session — used for the `projects` router) or `CsrfProtect` (checks the CSRF token but doesn't require a prior session — used for anonymous/magic-link/ORCID-adjacent and share-link-join endpoints, where a session may not exist yet). `GET /auth/csrf` primes the `csrftoken` cookie for the SPA to read before its first POST.

**Rate limiting (`core/ratelimit.py`):** fixed-window counter on Django's default cache. Applied to anonymous login, magic-link request (per-email and per-IP), and share-link join. Documented limitation: LocMemCache is per-process, so this only holds under a single worker (fine for `runserver`/dev; a multi-worker prod deployment would need a shared cache backend — not in the current stack).

**CORS:** `django-cors-headers` added, since Phase 2's web app will call the API cross-port in dev. `CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS`/`FRONTEND_URL` are env-driven (defaults to `http://localhost:5173`).

**Acceptance check (Plan.md §9, Phase 1):**
- 29 Django tests (`accounts`, `projects`, `health`), all passing against real Postgres, both via `manage.py test` and via `pytest` (the CI path) — covers all three sign-in paths, magic-link single-use/expiry, stranger-blocked-without-share-link, anonymous-joins-and-is-scoped-to-that-project-only, role enforcement (viewer can't rename, only owner can create/revoke share links), CSRF enforcement (rejected without token, accepted with matched cookie+header).

The callback code path is covered by a mocked unit test instead. ## Phase 3 — Compilation service (sandboxed, dual compiler): DONE

Implements Plan.md §7 in full. This is the security-critical core of the project, so every control was verified against real behavior, not assumed — including one piece of research (latexmk's actual flag semantics) fetched from primary docs before writing any code, per CLAUDE.md's rule for exactly this situation.

**`docker/texlive/`** (new) — the ephemeral sandbox worker image: Debian slim + `texlive-latex-base/-recommended/-extra`, `texlive-fonts-recommended`, `texlive-xetex`, `latexmk` (not `texlive-full`, which is ~5-7GB and unnecessary). **`apps/compile`** rewritten from the Phase 0 health stub into the actual orchestrator (`sandbox.py` + `main.py`). Verified against latexmk's real documentation before implementing, because memory was unreliable here and getting it wrong would silently leave shell-escape enabled: **latexmk has no native `-no-shell-escape` flag** — it must be baked directly into the `$pdflatex`/`$xelatex` command-override variable (`-e '$pdflatex=q/pdflatex -no-shell-escape .../'`), which is what `sandbox.py` does. Every §7 control is implemented and independently verified live (see below): no shell-escape, `--network none`, fresh container per job via the Docker Engine API (`cap_drop=["ALL"]`, `pids_limit`, `mem_limit`, `nano_cpus`, `read_only=True`, non-root `user=`, `no-new-privileges`), a wall-clock timeout enforced by `compile` itself (kills + removes the container on expiry, doesn't just abandon it), and a second independent path-safety check (`_safe_extract`) on top of the validation `projects/paths.py` already does, since untrusted-tar-becomes-filesystem-write is its own well-known vulnerability class (CVE-2007-4559-style) regardless of what already validated the logical paths upstream.

**Two real bugs found via testing, both fixed:**
1. **Docker-in-Docker sibling-container path bug.** `apps/compile` talks to the *host's* Docker daemon over the mounted socket to spawn job containers. `tempfile.mkdtemp()` was creating paths inside `compile`'s own private container filesystem — invisible to the host daemon, which resolves bind-mount sources against the *host* filesystem. Every job container therefore mounted an empty directory and reported "Could not find file 'main.tex'" despite the file genuinely existing on the `compile` side. Fixed by bind-mounting one real host directory (`./.compile-jobs`, via `${PWD}`) into `compile` at a fixed path, with `sandbox.py` writing job files there (visible to itself) while telling the *host* daemon the matching host-side path when specifying the sibling container's mounts. Documented in `sandbox.py` directly since it's exactly the kind of thing that's cheap to reintroduce.
2. My own first shell-escape test assertion was too weak to actually prove anything (an `or`-chain that was nearly always true). Replaced with real evidence: the compile log shows the exact `pdflatex -no-shell-escape ...` invocation and zero `runsystem` trace — TeX's documented behavior when shell-escape is disabled is to treat `\write18` as an inert `\write`, which is exactly what happened (compile succeeded normally, no command executed, no trace).

**`apps/api`**: new `ProjectSettings` (compiler, main_doc_path, central_bib_path — lazily created with defaults, matching Plan.md §8's schema which Phase 2 deliberately deferred) and `CompileRun` models. `projects/compile_api.py`: `GET`/`PATCH /projects/:id/settings`, `POST /projects/:id/compile` (materializes the project's current files from MinIO into an in-memory tar, dispatches to `compile`, stores the returned PDF/log/synctex back to MinIO, records a `CompileRun`), `GET /projects/:id/compile-runs[/​:id/pdf|/log]`. ## ORCID login 400 on user's production deployment (`freeleaf-api.deepmicroscopy.org`): RESOLVED

Root cause, found via the new logging (see below) within one round trip: `DisallowedHost` — the user's deployment routes the web app and the api to two *different* hostnames via their own reverse proxy (`freeleaf.deepmicroscopy.org` for the SPA, `freeleaf-api.deepmicroscopy.org` straight to the api), bypassing `apps/web/nginx.conf`'s same-origin proxying entirely. `DJANGO_ALLOWED_HOSTS` didn't include the api's own host, so Django rejected the request outright — not the session/cookie issue the vague original 400 suggested.

This is a valid, real-world deployment topology this repo's docs didn't cover — `.env.prod.example` and `docker-compose.prod.yml` only documented the same-origin-via-nginx setup, implicitly assuming `DJANGO_ALLOWED_HOSTS`/`FRONTEND_URL`/`CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS` are always the same one value. Fixed the docs, not just answered the user once:
- **`.env.prod.example`** now documents both topologies explicitly side by side: (A) same-origin via nginx — all four values identical; (B) split subdomains — `DJANGO_ALLOWED_HOSTS`/`ORCID_REDIRECT_URI` take the **api's** host, `FRONTEND_URL`/`CORS_ALLOWED_ORIGINS`/`CSRF_TRUSTED_ORIGINS` take the **web app's** host (they're no longer the same value once the SPA's fetch() calls are genuinely cross-origin).
- **README**'s production section links to that and flags the specific failure signature (400 with nothing useful in the browser → check `docker compose -f docker-compose.prod.yml logs api` for `DisallowedHost` first).

The error-message-splitting and `LOGGING` config added while diagnosing (see previous revision of this entry) stay — they're generally useful, not just for this one bug, and are what actually surfaced the real cause in one round trip instead of several rounds of guessing.

**Separate follow-up while deploying:** `VITE_API_ORIGIN` (what the web app's JS uses to reach the api — needed for the split-subdomain topology above) was hardcoded to empty string in `docker-compose.prod.yml`'s build args, not configurable at all. ## Split-subdomain deployment: "CSRF token missing" creating projects — RESOLVED

Follow-up to the `DisallowedHost` fix above: once ORCID login worked (session cookie confirmed round-tripping correctly in the logs), `POST /api/projects` still failed with Django's `Forbidden (CSRF token missing.)`. Root cause: `CSRF_COOKIE_DOMAIN` defaults to host-only, so the csrftoken cookie set by the api's own host was invisible to `document.cookie` on the *web app's* different host — the frontend's CSRF middleware (`packages/shared/src/client.ts`) had nothing to echo back as `X-CSRFToken`. The session cookie didn't have this problem (subdomains are same-site, so the browser resends it automatically regardless of what JS can read) — only the CSRF double-submit pattern needs JS to actually read the cookie value, which is where host-only scoping bit.

Fix: new `CSRF_COOKIE_DOMAIN` setting (`config/settings.py`, `docker-compose.prod.yml`), set to the parent domain shared by both hosts (with a leading dot) — e.g. `.freeleaf.deepmicroscopy.org` for the user's actual `freeleaf.deepmicroscopy.org` / `api.freeleaf.deepmicroscopy.org` split. `.env.prod.example`'s topology-B section rewritten to use a sub-subdomain example (`api.freeleaf.example.com`) instead of a sibling one, since that's what the user ended up deploying and it makes the `CSRF_COOKIE_DOMAIN` value obviously-correct by construction. Attempted a local repro via fake `/etc/hosts` subdomains before the user settled on their final domain shape; abandoned that in favor of just shipping the fix once the topology was confirmed. **User confirmed working in production.**

## Phase 5 — Real-time collaboration: DONE

Yjs (§6, Node `collab` service — decision recorded in `docs/decisions/collab.md` back in Phase 0) now backs `.tex` editing: multi-cursor, live presence (including anonymous users), reconnect-after-drop convergence, and compiles reflecting merged state.

**Auth model.** `collab` has no DB/session access of its own by design (keeps Node confined to one service, per the ADR). `GET /projects/:id/files/:id/collab-token` (session-authed, reuses the existing membership/role check) mints a 60s HMAC-signed token (`core/collab_tokens.py` ↔ `apps/collab/src/token.ts` — same "`base64url(json).hex-hmac-sha256`" format on both ends, deliberately not JWT: one shared secret, one fixed shape, nothing to verify-against-live-docs since there's no library involved). The frontend passes it as a WS query param; `collab` verifies the signature+expiry locally, no round trip to `api` needed per connection. A second, separate surface — `/api/internal/collab/files/:id/content` (GET/PUT) — lets `collab` read/seed a room's initial content and periodically persist the merged plaintext back through api's existing MinIO-backed storage layer, protected by a shared-secret header (webhook-style trust model, not session auth, since there's no per-request user identity to check there).

**Wire protocol.** `y-websocket` v3 turned out to be client-only now (dropped its bundled server — a real, version-specific finding, not an assumption carried over from an older version). Hand-rolled the server side directly on `y-protocols/sync` + `y-protocols/awareness` + `lib0`, matching the browser client's exact outer message-type framing (verified by reading the installed client's own source, not from memory). Viewers get a read-only room: their sync step2/update messages are accepted-but-ignored server-side (in addition to the existing REST-layer role gate), so the WS path can't be used to bypass viewer restrictions.

**Persistence.** Each room seeds its `Y.Text` from the file's current stored content on first join (empty room → new `Room`), and flushes plaintext back every 4s if dirty, plus once more on the last client leaving (`apps/collab/src/room.ts`). No separate Yjs-binary-state storage — reconstructing a fresh `Y.Doc` from the last-known plaintext on next open was a deliberate simplification over persisting raw CRDT history, acceptable since content (not edit-history/undo-stack) is the thing that must never be lost across a fully-empty-room gap.

**Frontend.** `y-codemirror.next`'s `yCollab` extension binds directly to the shared `Y.Text`, giving sync + remote-cursor rendering (with name/color labels) for free. `CodeMirrorEditor` no longer does its own explicit save — content syncs continuously; the debounced "auto-compile on save" behavior from Phase 4 is now driven off `ydoc.on('update', ...)`, debounced 1500ms, only armed *after* the initial content sync (so just opening a file doesn't itself trigger a compile). A small presence row (colored initials, one per other connected user) sits next to a Live/Connecting/Reconnecting status pill.

**Three real bugs found and fixed during this pass** (all via live testing, not caught by `tsc`):
1. **Frontend forgot to regenerate the typed OpenAPI client** after adding the new `collab-token` endpoint — surfaced as a real prod build failure (`tsc` error on `api.GET(...)` not recognizing the new path) the user hit directly. Fixed by regenerating `packages/shared/src/generated.ts`, per this repo's own convention of generating types rather than hand-writing them (an in-flight hand-rolled `fetch()` workaround was reverted in favor of the generated client).
2. **`PDFDocumentProxy` doesn't have a `destroy()` method** (unrelated pre-existing Phase 4 code, caught mid-session while debugging something else) — it's on `PDFDocumentLoadingTask`. Fixed by holding onto the loading task itself.
3. **A genuine race in the hand-rolled collab server**: the browser's Yjs client sends its initial sync step1 synchronously on WS open, but the server only attached its `message` listener *after* `getOrCreateRoom`'s async content-fetch resolved — so that first message was silently dropped (Node's EventEmitter doesn't queue events for not-yet-attached listeners), the server never replied with step2, and the client never marked itself synced. Diagnosed with a minimal Node WS client script logging raw bytes on both ends (confirmed well-formed bytes were arriving; the gap was purely the missed first message) before fixing it: the message listener now attaches synchronously at connection time and buffers into a queue until the room is ready, then drains the queue before switching to live processing.
   - Also fixed along the way: a malformed reply where the server's sync-message handler always pre-wrote the outer message-type byte before checking whether there was anything to reply with, sending 1-byte non-replies for `update`/`step2` messages that a real client couldn't parse (`Unexpected end of array`).
   - And a `DJANGO_ALLOWED_HOSTS` gap identical in shape to the earlier ORCID production bug: `collab` calls `api` using the Docker-network hostname `api`, which wasn't in the allowed-hosts list, so every seed/persist call 400'd as `DisallowedHost`. Added `api` to both dev and prod defaults, documented in `.env.prod.example` alongside the existing topology notes (needed regardless of same-origin vs split-subdomain topology, since it's an internal Docker-network call either way).

**Live verification** (Playwright, two independent browser contexts — an invited editor via magic link, and a second anonymous "guest" joining the same share link, since anonymous-via-invite is exactly the real usage path per the auth model): both land on `main.tex`; each sees the other's presence dot; edits typed in one appear live in the other with no save/reload; dropping one client's connection and having the other keep editing, then reconnecting the dropped one, converges to all edits present with no data loss; clicking Recompile after all edits produces a successful compile. Screenshots + script: `pw/drive_phase5.mjs`.

**Known limitations, called out rather than silently deferred:** no automated test coverage for `apps/collab` itself (verified live only — Node has no test setup in this repo yet); if multiple clients are typing when the debounce settles simultaneously, more than one may independently trigger a compile (wasteful, not incorrect); a maliciously crafted step2 message could theoretically still slip a viewer's edit through the WS-layer read-only gate (the real enforcement is the REST API's role check, this is defense-in-depth only, noted in `server.ts`).

## SyncTeX click-to-source (both ways) + PDF zoom: DONE

**Approach.** Deliberately used the real `synctex` CLI (`texlive-binaries`, added to `apps/compile/Dockerfile`) rather than hand-parsing the binary/gzipped `.synctex.gz` format — that format's coordinates are relative and accumulate through a nested box hierarchy, and a naive from-scratch parser risks silently *wrong* jumps, which is worse than no feature at all. New `apps/compile/synctex_query.py` (`forward_search`/`backward_search`, shelling out to `synctex view`/`synctex edit`) and two new `apps/compile` HTTP routes (`/synctex/forward`, `/synctex/backward`), dispatched from new `apps/api` endpoints `GET .../compile-runs/{id}/synctex/forward` and `.../synctex/backward` (fetch `pdf_key`/`synctex_key` from storage, base64 to the compile service, same pattern as the existing `dispatch_compile`). 6 new backend tests mocking `_dispatch_synctex`, all passing; OpenAPI client regenerated.

**Real bug found and fixed during live verification: highlight/click coordinates were vertically wrong.** First implementation used PDF.js's `viewport.convertToPdfPoint`/`convertToViewportPoint`, which assume native PDF user-space (origin bottom-left, y up). Screenshotted verification showed the forward-search highlight box landing in empty space, nowhere near the clicked line — SyncTeX's `h`/`v`/`x`/`y` are actually already in **top-left-origin, y-down** page coordinates (matching TeX's internal box/DVI convention, not the PDF spec), so PDF.js's flip was introducing a large, wrong vertical offset. Fixed by replacing both conversions with a plain scale multiply (no axis flip) in `PdfViewer.tsx`. Confirms why the earlier compile-service-level round-trip test (forward search's raw output fed straight back into backward search) passed even before this fix — it never touched the PDF.js viewport math, only the CLI's own self-consistent coordinates.

**Frontend wiring:** `PdfViewer.tsx` — click on rendered PDF text → backward search → `onSourceClick(page, x, y)`; a temporary highlight box + `scrollToPosition()` imperative handle for forward search, both driven off `currentScaleRef` (the actual render scale) rather than PDF.js viewport transforms. `CodeMirrorEditor.tsx` — Cmd/Ctrl+click on editor text → `onJumpToPdf(line)`; a `jumpTarget` prop applies a pending cursor jump once the Yjs-synced view is ready (handles both same-file jumps and cross-file jumps where the view remounts). `CompilePane.tsx` exposes `jumpToPdf()` on its imperative handle and resolves backward-search results via the current compile run's id. `EditorTab.tsx` wires the two directions together, including switching the selected file when a PDF click resolves to a different file (e.g. jumping into a `.bbl`-adjacent citation actually defined in another `.tex` file).

**PDF zoom (requested alongside click-to-source).** Zoom in/out/reset-to-fit-width toolbar in `PdfViewer.tsx`, re-rendering all page canvases at the new scale (pages/document stay loaded — only `page.render()` reruns, no refetch). A `renderTokenRef` guards against overlapping renders from rapid zoom clicks.

## Warnings-only-from-last-run fix: DONE

**Real bug, exactly as flagged in PLAN.md §9's Phase 8 notes: "unknown citations will be shown as warnings for every compile."** Root cause: our captured "log" is latexmk's stdout for *every* internal rerun of the engine concatenated together (`apps/compile/sandbox.py` captures `docker logs`, not the single-run `.log` file) — latexmk reruns pdflatex/xelatex multiple times per compile to resolve citations/references (e.g. run → bibtex → rerun → rerun until stable), and `log_parser.parse_log()` was scanning the *entire* concatenated transcript, so a "Citation undefined" warning from an early pass surfaced even once a later pass in the *same compile* resolved it. Fixed with `_last_engine_run()`: each engine invocation announces itself with a `This is pdfTeX/XeTeX/LuaTeX, Version ...` banner, so only the segment after the *last* banner (the compile's authoritative final state) is parsed. New fixture/test (`MULTI_RUN_RESOLVED_CITATION_LOG`) reproduces the exact scenario — a citation undefined in pass 1, resolved by bibtex + pass 2 — and confirms zero warnings, not one stale one. All 29 `log_parser`/`compile` tests pass; verified the regex against a real multi-run log sample too.

## Version history / snapshots + Time Travel UI: DONE

**Model.** New `ProjectSnapshot` (`kind`: auto/manual, `label`, `description`, `created_by`, `archive_key`, `content_hash`) — each snapshot is a zip of every file's content at that moment, reusing the exact same format as `/export` (extracted `build_project_zip_bytes()` out of `export_project_zip` so both share it). A full-copy-per-snapshot is deliberately simple over per-file deltas — LaTeX projects are small, and it means diffing or restoring one file is just "extract one zip entry," no delta-reconstruction logic needed.

**New `apps/api/projects/versions_api.py`:** `POST /projects/{id}/snapshots` (manual or auto, editor+; auto-snapshots dedupe against the most recent snapshot's `content_hash` so an inactivity timer firing with nothing new to capture is a no-op), `GET /projects/{id}/snapshots` (list, any member), `GET .../snapshots/{id}/file-content?path=...` (any member, for diffing), `POST .../snapshots/{id}/restore` (editor+). 17 new backend tests, all passing.

**Restore semantics: a genuine full revert, not a merge.** Existing files are overwritten to match the target snapshot; files deleted since the snapshot are recreated; files added after the snapshot are deleted outright — "restore" means "make the project exactly like this version," matching user expectations for time-travel. Because that's destructive, restore **always takes a safety snapshot of the current state first** (kind=`auto`, labeled with the restore-to date) — nothing is ever actually lost, since undoing a restore is just another restore.

**Real design problem worked through: live-collaboration-aware restore.** A naive restore that just overwrites storage would get silently clobbered by an open Yjs room's next periodic flush (Phase 5), or would leave a currently-editing collaborator's view stale. Added a new `apps/collab` endpoint (`POST /replace/:fileId`, shared-secret protected, mirroring the existing `/flush/:fileId`) that applies a *real* Yjs delete+insert transaction when a room is open — so it merges and broadcasts to connected clients exactly like a normal edit would — and falls back to the same direct `persistFileContent()` write `/flush` already uses when no room is open. The API's restore endpoint calls this for `.tex`/`.bib` files and falls back to a direct storage write if collab is unreachable (best-effort, matching `flush_collab_rooms`'s established resilience pattern), so restore still functionally succeeds even with collab down.

**Frontend:** new "History" tab (alongside Editor/Library/Settings) — chose a tab over PLAN.md's literal "toggleable sidebar replacing the review pane" wording since the review pane doesn't exist yet (Phase 8's editing-modes/comments work, not started) and a tab fits the existing information architecture without inventing a new UI mechanism prematurely; noting this as the deliberate deviation. `HistoryTab.tsx`: snapshot list (named vs automatic badge, label/description/timestamp/author) + "Save a version" inline form; selecting a snapshot shows a side-by-side diff (new `DiffView.tsx`, using the `diff` package's `diffLines` — added as a new dependency, justified since Phase 8 explicitly specs "side-by-side split file diff" and hand-rolling a line-diff algorithm is exactly the kind of correctness risk not worth taking) between that version and the file's current live content, plus a "Restore to this version" button behind a confirmation dialog explaining the safety-snapshot guarantee.

**Automated snapshots.** Implemented client-side (no job queue/cron exists in this stack, and PLAN.md doesn't call for adding one): `EditorTab.tsx` tracks real keydown events (1000 keystrokes → immediate auto-snapshot) and resets a 5-minute inactivity timer on every Yjs update (fires → auto-snapshot), with a 60s client-side minimum gap between auto-snapshots as a defensive floor. Only wired when `canWrite` — viewers never trigger snapshot attempts.

Verified live end-to-end via Playwright against a real owner session: created a project, typed content, saved a named version ("First draft"), edited further, opened the diff view and confirmed the added paragraph rendered correctly aligned and highlighted on only the "Current" side, clicked "Restore to this version," confirmed the safety-snapshot dialog copy, restored, and confirmed both that a new "Automatic snapshot" entry appeared (with the correct "Automatic backup before restoring to ..." description) and that the editor's live content reverted to exactly the saved version — including diffing the restored state against itself afterward and confirming "No differences from the current file content."

## Prod build fix: stale lockfile + a latent type error `tsc --noEmit` was silently missing: RESOLVED

**Bug report:** production `web` build failing with `ERR_PNPM_OUTDATED_LOCKFILE` on `pnpm install --frozen-lockfile`.

**Root cause #1:** `pnpm-lock.yaml` lives at the repo root, which isn't bind-mounted into the `web` dev container (only `apps/web` and `packages/shared` are). Adding the `diff` dependency (for the version-history diff view) via `pnpm add diff` inside the container updated `apps/web/package.json` (bind-mounted, persisted to host) but the lockfile write happened in the container's own copy of the repo root and never reached the host — leaving `pnpm-lock.yaml` silently stale. Fixed with `pnpm install` from the host repo root (host `pnpm@9.15.0` matches root `package.json`'s `packageManager` exactly); verified with `pnpm install --frozen-lockfile`.

**Root cause #2, uncovered by fixing #1:** the real prod build (`tsc -b && vite build`) then failed on an actual type error in `EditorTab.tsx` (a `POST /snapshots` call missing required `label`/`description` fields) that had been present the whole time. It was invisible because `apps/web/tsconfig.json` has `"files": []` and only `references` — a valid TS project-references setup, but it means plain `tsc --noEmit` (what this session had been "verifying" with after every change) type-checks *zero files* and always exits 0. Fixed the actual type error; going forward, real verification needs `tsc -b`, not `tsc --noEmit` — saved to memory.

**Found and fixed while investigating:** the `collab` service's Docker image was stale — new version-history code (`/replace/:fileId`) had been added to its source earlier in this session but the container was never rebuilt (it's not bind-mounted, unlike `web`/`api`), so the endpoint didn't exist in the running service at all. The earlier "verified" restore test had passed only because it incidentally exercised a fallback code path, not the actual new logic. Rebuilt (`docker compose build collab && docker compose up -d collab`) and re-verified the live-collaboration-aware restore path specifically — with a genuine two-browser-context Playwright test (one context keeps a file open in the editor the whole time; a second context triggers the restore) confirming the open editor updates in real time via the real Yjs transaction, not just on reload.

All three fixes verified: `pnpm install --frozen-lockfile` succeeds, a full `docker build -f apps/web/Dockerfile.prod .` succeeds end-to-end, and the two-context live-restore test passes.

## Comments pane (add/reply/resolve): DONE

**Model.** New `Comment` (project + file FKs, `anchor_line`, `body`, `created_by`, `resolved`/`resolved_by`/`resolved_at`, self-referencing `parent`).
